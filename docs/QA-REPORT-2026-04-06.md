**Files:**

**Fix:** Standardize on one claim name (`userId`) across all token generation and test helpers.

## Security Issues

### SEC-001: Mass Assignment Vulnerability in Website Controller
**File:** `websitesaas/backend/src/controllers/website.controller.js:72`  
**Severity:** High

```js
data: req.body  // User can update ANY field including userId, status
```

A malicious user could send `{ status: "PUBLISHED", userId: "other-user" }` and potentially take ownership of websites or manipulate status.

**Fix:** Whitelist allowed fields:
```js
data: pick(req.body, ['name', 'slug', 'config', 'templateId'])
```

### SEC-002: Stripe Webhook Body Parsing Order Conflict
**File:** `websitesaas/backend/src/index.js:39-40`, `payments.routes.js:13`  
**Severity:** High

The global `express.json()` middleware at `index.js:39` consumes the request body BEFORE it reaches `payments.routes.js:13` which declares `express.raw({ type: 'application/json' })`. Stripe webhook signature verification (`stripe.webhooks.constructEvent`) requires the raw body to validate the HMAC signature — this will always fail.

**Fix:** Move payment routes BEFORE `express.json()`, or use a route-specific body parser:
```js
app.use('/api/payments/webhook', express.raw({ type: 'application/json' }), paymentRoutes.webhook)
app.use(express.json())  // after
```

### SEC-003: Default JWT Secrets in Production
**File:** `backend/src/utils/jwt.js:3-4`  
**Severity:** Medium

```js
const JWT_SECRET = process.env.JWT_SECRET || 'your-secret-key'
const JWT_REFRESH_SECRET = process.env.JWT_REFRESH_SECRET || 'your-refresh-secret'
```

If environment variables are not set, trivially guessable defaults are used.

**Fix:** Throw on startup if secrets not configured in production:
```js
if (process.env.NODE_ENV === 'production' && (!JWT_SECRET || JWT_SECRET === 'your-secret-key')) {
  throw new Error('JWT_SECRET must be set in production')
}
```

### SEC-004: Password Reset Token Exposure Risk
**File:** `backend/src/controllers/email.controller.js:26`

The password reset URL embeds the token in the URL query string which may be logged in browser history, proxy logs, and referrer headers.

**Recommendation:** Consider using a one-time link approach with a POST endpoint and short-lived session token.

### SEC-005: Insufficient Password Strength Validation
**File:** `backend/src/middleware/validation.js:5`

```js
password: z.string().min(8, 'Password must be at least 8 characters')
```

Allows passwords like "aaaaaaaa" which meet length but have zero entropy.

**Recommendation:** Add `z.string().min(8).regex(/[A-Z]/)` etc., or use `zxcvbn` library.

## Code Quality Issues

### CQ-001: Prisma Client Connection Pool Exhaustion Risk
**Files:** `email.controller.js:5`, `website.controller.js:2`, and likely others

Each controller does `new PrismaClient()` creating separate connection pools.

**Fix:** Create a single PrismaClient singleton and export it:
```js
// prisma/client.js
const { PrismaClient } = require('@prisma/client')
const prisma = new PrismaClient()
module.exports = prisma
```

### CQ-002: Async Fire-and-Forget Email in Auth Controller
**File:** `backend/src/controllers/auth.controller.js:25-26`

```js
emailService.sendWelcomeEmail(user)  // No await
emailService.sendAdminNewUserAlert(user)  // No await
```

`sendWelcomeEmail` creates DB records (`emailLog.create`). If this fails after user is created, the DB state is inconsistent and the error is silently swallowed.

**Fix:** Either await or properly handle errors, or move to a job queue.

### CQ-003: Duplicate Email Route Endpoints
**Files:** `email.routes.js:8-10`, `auth.routes.js:13-15`

Endpoints like `forgot-password`, `reset-password`, `verify-email` are defined on BOTH `/api/auth/*` and `/api/email/*`. This is confusing and creates maintenance burden.

**Fix:** Pick one canonical route for each auth-related email endpoint.

### CQ-004: Email Routes Not Authenticated But Mutate State
**File:** `email.routes.js:8-10`

The `/api/email/reset-password` and other endpoints are public but mutate password reset tokens and email verification status. These should not share the `/api/email` route path since they're auth endpoints.

### Failing Tests Summary

| Suite | Test | Issue |
|-------|------|-------|
| Auth API | POST /api/auth/register (1 subtest) | Module cache / mock isolation — `Module.prototype.require` monkeypatch leaks between tests |
| Auth API | POST /api/auth/refresh-token (1 subtest) | Token payload mismatch: `{ sub }` vs `{ userId }` |
| Template API | Admin Write Access (1 subtest) | Module cache not fully cleared for auth middleware |
| Website API | GET /api/websites (1 subtest) | Same mock isolation issue |
| Website API | POST /api/websites (1 subtest) | Mock isolation issue |
| Website API | PUT /api/websites/:id (3 subtests) | Mock isolation + route reuse |
| Website API | DELETE /api/websites/:id (2 subtests) | Mock isolation issue |
| Website API | POST /auth/websites/:id/publish (1 subtest) | Expected 200 got 401 — mock not applied due to cache |

**Root Cause:** The `Module.prototype.require` monkeypatching approach for Prisma mocking doesn't work reliably with Node.js `require` caching. Routes cached from previous test runs use the real PrismaClient. The `clearRouteCache()` functions in individual test files don't clear the full dependency graph.

**Recommendation:** Replace with proper dependency injection or use `mock` from `node:test` (built-in mocking) for clean test isolation. This is the highest-impact improvement for the test suite.
